# Diary: Vault Deployment Project

## 2026-03-11

### Наблюдения
- Пользовательский запрос фокусируется на полном bootstrap с минимальным числом итераций.
- Базовые вводные известны: VMware 7.0.3, RedOS 8, сеть уже настроена, остальное требуется автоматизировать.
- Для production-grade архитектуры критичны уточнения по TLS/PKI, политике unseal, требованиям к доступности и интеграциям.

### Решения
- Зафиксирована целевая архитектура Vault с RAFT и минимум 3 узлами.
- Принято решение начать с полноценного Discovery-блока (вопросы вынесены в `qa.md`).
- Для управляемости проекта введен трекер задач с приоритетами и статусами в `Tasktracker.md`.

### Проблемы
- На текущем этапе отсутствуют детальные инфраструктурные параметры (CPU/RAM/disk/FQDN/сегменты сети).
- Не определена модель TLS и процесс инициализации/unseal.
- Не подтверждены операционные требования (backup/DR/SLA/monitoring).

## 2026-03-11 (этап Discovery: сбор ответов)

### Наблюдения
- Подтвержден переход к следующему шагу: сбор параметров из `qa.md`.
- Для минимизации числа проходов нужен структурированный ответ одним блоком.

### Решения
- Добавлена отдельная задача в `Tasktracker.md` на закрытие полного опросника.
- Подготовлен приоритетный список параметров, без которых нельзя безопасно начать написание ролей.

### Проблемы
- Пока отсутствуют фактические данные по PKI/TLS и по модели unseal, что блокирует проектирование bootstrap-ролей.

## 2026-03-11 (этап Discovery: ответы заказчика получены)

### Наблюдения
- Получен структурированный набор параметров по топологии, сетям, TLS и bootstrap-модели.
- Архитектурно подтвержден стек: 3-node Vault RAFT, HAProxy VIP, Shamir manual unseal, AppRole + KV v2.
- Часть эксплуатационных параметров осталась неуточненной (RPO/RTO, snapshots, monitoring, owner team).

### Решения
- `Project.md` актуализирован фактическими параметрами среды.
- `qa.md` разделен на закрытые пункты и оставшиеся блокеры.
- В `Tasktracker.md` добавлены точечные задачи на закрытие governance/operations вопросов.

### Проблемы
- Не определены `key_custodians` и политика initial root token.
- Не определены обязательные приемочные тесты и версия Ansible.
- Отсутствуют требования по snapshot/DR и monitoring stack.

## 2026-03-11 (этап реализации: каркас Ansible создан)

### Наблюдения
- Созданы inventory для `dev/stage/prod`, playbooks и каркасы ролей.
- Добавлены стартовые конфигурации качества: `ansible.cfg`, `.ansible-lint`, `.yamllint`, `.pre-commit-config.yaml`.
- YAML-файлы проходят синтаксический парсинг.

### Проблемы

## 2026-03-11 (этап реализации: hardening ролей, итерация 1)

### Наблюдения
- Базовый каркас ролей требовал усиления по firewall и проверке TLS-артефактов.
- Для `shamir_manual` важно оставить bootstrap безопасным по умолчанию без автоматического раскрытия ключей.

### Решения
- В `vault_config` добавлены проверки наличия TLS-файлов, нормализация прав и strict firewall-модель по CIDR для портов `8200/8201`.
- В `os_baseline` добавлено управление SSH-правилами firewalld по разрешенным подсетям.
- В `vault_install` добавлен checksum-aware download и удаление временных артефактов.
- В `vault_bootstrap` усилен workflow status/init/unseal и включение audit-device при наличии root token.

### Проблемы
- Проверка `ansible-playbook --syntax-check` по-прежнему недоступна в текущем окружении из-за отсутствия Ansible.

## 2026-03-11 (этап реализации: результаты итерации 2)

### Наблюдения
- В ролях оставались риски по корректной передаче firewall rich-rules и по согласованию bootstrap с root token policy.
- Для передачи в эксплуатацию потребовался отдельный runbook с практическими процедурами.

### Решения

### Проблемы

## 2026-03-11 (этап реализации: единый URL и локальный TLS-файл)

### Наблюдения
- Появилось дополнительное требование: доступ к UI через единый URL в браузере.
- Требуется использовать единый TLS-файл, который оператор кладет в каталог проекта с плейбуком.

### Решения
- Запущена отдельная задача на доработку inventory/roles под `vault_public_url`.
- Принято решение добавить controller-side доставку TLS-артефактов из каталога проекта.

### Проблемы
- Нужно аккуратно обеспечить совместимость с текущей моделью ручной доставки TLS.

## 2026-03-11 (этап реализации: единый URL и TLS из репозитория реализованы)

### Наблюдения
- Для доступа через браузер нужен стабильный URL, не зависящий от конкретной ноды.
- Для операционной простоты востребован режим единого TLS-файла (`vault.pem`) в каталоге проекта.

### Решения
- Введен `vault_public_url` как единая точка входа UI/API через LB.
- API firewall ограничен LB-источником (`10.255.107.49/32`) в `prod`.
- Реализована controller-side доставка TLS (`bundle`/`split`) с проверками наличия файлов на control-host.
- Добавлены `tls/README.md` и `.gitignore` для безопасной работы с TLS-артефактами.

### Проблемы
- При отсутствии SNAT на LB может потребоваться расширить `vault_allowed_api_cidrs` под реальный source IP.
